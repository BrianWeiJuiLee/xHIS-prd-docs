## 1. Repo 用途與邊界

**這個 repo 是什麼：** xHIS 產品團隊的正式 PRD（Product Requirements Document）存檔空間。main branch 上非範例的 PRD 文件（`is_example: false`），是研發團隊實作的唯一授權依據。

**這個 repo 不是什麼：** 不儲存設計稿、API spec、程式碼、測試腳本，或任何研發產出物。

**主要使用者：** xHIS PM 團隊（約 10 人），每人負責一至多個系統的 PRD。研發工程師為唯讀閱覽者。

## 2. xHIS 產品背景

**xHIS（Extended Hospital Information System）** 是一套為台灣醫療機構設計的整合型醫院資訊系統，涵蓋從病患到院到出院的完整就醫流程，並介接台灣全民健康保險（健保）相關業務流程。

**部署對象：** 台灣中大型醫院（區域醫院至醫學中心等級）

**監管環境：**
- 醫療法（Medical Care Act）：病歷資料需保存 7 年；電子病歷需電子簽章
- 全民健康保險法（National Health Insurance Act）：健保費用申報格式與時程由健保署規定
- 個人資料保護法（Personal Data Protection Act）：病患資料存取需記錄稽核日誌
- JCI（Joint Commission International）認證標準：護理文件完整性要求

## 3. 系統架構總覽

xHIS 由 11 個子系統組成，各自獨立部署但透過 API 整合。

| 代碼 | 中文名稱 | 系統職責邊界 |
|-----|---------|------------|
| **Register** | 掛號系統 | 病患身份識別、門診/急診/住院掛號、預約管理、候診管理 |
| **Billing** | 批價系統 | 門診/急診/住院費用計算、健保與自費拆算、部分負擔、收費開立、退費 |
| **Declare** | 申報系統 | 健保費用月結申報（D-file 格式）、核退件追蹤、補申報作業 |
| **Referral** | 轉診系統 | 電子轉診單開立、健保署雙向轉診平台介接、轉診狀態追蹤 |
| **Pharmacy** | 藥局系統 | 門診/急診/住院調劑、掃碼核對、庫存扣除、發藥叫號、退藥 |
| **OPD** | 門診醫令系統 | 門診診間醫令開立（藥品/檢驗/處置）、Order Set 管理、CDSS 整合 |
| **ER** | 急診醫令系統 | 急診醫令開立、急診看板、留觀管理 |
| **IPD** | 住院醫令系統 | 住院長期/臨時醫令、入院開單範本、轉科/出院醫令、電子簽章 |
| **ERNIS** | 急診護理系統 | 急診檢傷分類（TTAS）、急診護理評估與記錄、醫令執行確認 |
| **IPDNIS** | 住院護理系統 | 入院護理評估、護理計劃（NANDA）、SOAP 護理記錄、交班摘要 |
| **Portal** | 首頁系統 | 角色化工作清單、通知中心、跨子系統快捷入口 |

### 系統邊界原則

- Register 負責「病患到院前」的身份與排程；OPD/ER/IPD 負責「診間內」的醫療作業
- 費用資料流：OPD/ER/IPD 醫令 → Billing 批價 → Declare 申報（單向，不可逆）
- 護理系統（ERNIS/IPDNIS）讀取醫令系統（ER/IPD）的醫令，執行後回寫確認狀態
- Portal 只讀取各子系統資料，不持有任何業務資料

## 4. 跨系統資料流與主要整合點

```
病患到院
   │
Register ──掛號完成──▶ OPD/ER（候診清單）
                              │
                     醫師開立醫令
                              │
              ┌───────────────┼───────────────┐
              ▼               ▼               ▼
          Pharmacy         檢驗系統         IPDNIS/ERNIS
         （調劑）          （採檢）         （護理執行）
              │
              └───▶ Billing（費用計算）──▶ Declare（申報）
```

**關鍵整合規則（所有 PRD 必須遵守）：**

1. **醫令送出不可撤銷**：OPD/ER/IPD 醫令送出後，不得刪除，只能開立停用或更正醫令
2. **批價完成不可刪除**：Billing 批價記錄不可刪除，退費需走退費流程並留存原始紀錄
3. **護理記錄不可刪除**：ERNIS/IPDNIS 的護理記錄一旦送出不可修改或刪除，更正需新建「更正記錄」
4. **申報資料為單向流**：Declare 系統從 Billing 讀取，不寫回 Billing；申報一旦送件，補申報需走獨立流程
5. **所有病患資料操作需寫入稽核日誌（Audit Log）**，記錄操作者、時間戳記、操作類型

| 申報人員 | 醫院行政 | Declare | 執行月結送件需主管授權 |

| 急診醫師 | 醫療 | ER, Referral | 醫令需本人電子簽章 |
| 主治醫師（住院） | 醫療 | IPD, Referral | 入院醫令需本人電子簽章 |

| 急診檢傷護理師 | 護理 | ERNIS | 具備 TTAS 分級輸入權限 |

| 病房護理師 | 護理 | IPDNIS | 具備護理計劃建立與記錄 |

## 7. 台灣醫療資訊關鍵術語

| 術語 | 說明 |
|-----|-----|
| 健保（NHI） | 全民健康保險，由健保署管理，醫院需定期申報醫療費用以獲給付 |
| 點數 | 健保給付計算單位；每點對應的實際金額每季浮動（點值） |
| 部分負擔 | 病患自行負擔的就醫費用比例（如門診掛號費、藥品部分負擔） |
| 重大傷病 | 健保法定免部分負擔疾病類別，病患持有效重大傷病卡可免部分負擔 |
| ICD-10-CM | 國際疾病分類第十版，台灣健保申報使用的診斷編碼系統 |
| TTAS | 台灣檢傷分類系統（Taiwan Triage and Acuity Scale），急診五級分類標準 |
| NANDA | 北美護理診斷協會診斷系統，台灣住院護理計劃的標準護理診斷來源 |
| GCS | Glasgow Coma Scale，意識狀態評估（E+V+M，3–15分） |
| NRS | Numeric Rating Scale，數字疼痛評估量表（0–10） |
| Morse Fall Scale | 跌倒風險評估工具，住院及急診護理常規使用 |
| 電子簽章 | 依醫師電子簽章法規，醫令需由開立醫師本人電子簽章方具效力 |
| 稽核日誌 | 所有涉及病患資料的系統操作記錄，需永久保存 |
| D-file | 健保署規定的電子申報資料格式，每月送件 |
| JCI | Joint Commission International 評鑑，護理文件完整性是主要評核項目 |
| 診別 | 初診（首次就診或該科 365 天內無紀錄）/ 複診（該科 365 天內有就診紀錄） |
| Order Set | 醫師自訂的常用醫令組合，可一鍵帶入節省開立時間 |

## 8. PRD 文件規範

### 8.1 Frontmatter Schema（所有欄位均為必填）

```yaml
---
title: string                # 功能名稱，範例文件需加【範例】前綴
system: string               # 格式："{代碼} {中文名稱}"，例如 "Register 掛號系統"
version: string              # 格式："主版號.次版號"，例如 "1.0"、"1.2"、"2.0"
author: string               # PM 姓名
created: string              # 格式：YYYY-MM-DD
updated: string              # 格式：YYYY-MM-DD（每次版本更新時修改）
status: string               # 限定值見下方 8.2 節
is_example: boolean          # true = 範例文件（不得作為研發依據）；false = 正式文件
---
```

### 8.2 Status 允許值（嚴格枚舉，不允許其他值）

| status 值 | 意義 |
|-----------|-----|
| `編輯中` | PRD 正在撰寫或修訂，尚未完成 |
| `內部審核通過` | 已通過 PM 團隊內部審查，內容凍結 |
| `外部審核通過` | 已通過客戶或主管機關確認，可進入實作 |
| `實作中` | 研發團隊已接手，正在開發中 |
| `已上線` | 功能已部署至正式環境 |

### 8.3 PRD 六節架構（所有節均為必填）

1. **Change History** — 版本異動紀錄表，每次 commit 前必須更新
2. **Requirement Overview** — 背景目的（2.1）、目標與範疇（2.2）、目標使用者（2.3）、非功能需求（2.4）
3. **Business Flow Overview** — Mermaid flowchart TD 流程圖（3.1）、步驟說明（3.2）、跨系統互動（3.3）
4. **Data Flow Overview** — Mermaid flowchart LR 資料流程圖（4.1）、關鍵資料項目（4.2）、API 規格（4.3）
5. **Use Cases** — 每個 UC 含：操作流程（5.x.1）、UI 畫面（5.x.2）、欄位規格與業務規則（5.x.3）、例外處理
6. **Test Cases** — 驗收測試表，含 P0/P1/P2 優先級

### 8.4 檔案命名規則

- 格式：`PRD-[FeatureName].md`（FeatureName 使用 PascalCase 英文）
- 例：`PRD-PatientSearch.md`、`PRD-OrderEntry.md`
- 放置位置：對應系統的資料夾（例如 `Register/PRD-PatientSearch.md`）

### 8.5 版本號規則

- **次版號（+0.1）**：局部修正、補充說明、錯字修訂
- **主版號（+1.0）**：需求範疇異動、Use Cases 新增或刪除、業務規則重大變更

## 9. AI 工作準則

### 9.1 閱讀規則

- `is_example: true` 的文件是**範例參考**，代表正確的格式與寫法，但其中的業務需求不具效力
- `is_example: false` 且 `status` 不為 `編輯中` 的文件，是當前最權威的需求定義
- 系統 README.md 內的 PRD 清單是該系統所有 PRD 的索引

### 9.2 生成 PRD 時

- 先讀取同系統資料夾下的現有 PRD，確保術語、角色稱呼、系統互動描述與現有文件一致
- 不確定的業務規則或技術細節，使用 `（待確認）` 標示，**不得自行推測或捏造**
- 業務規則（BR）必須有明確來源（法規規定 / 客戶需求 / 系統設計決策），若來源不明需標示
- 生成的 PRD 預設 `is_example: false`、`status: "編輯中"`
- 任何涉及刪除病患資料、醫令、護理記錄的需求，**必須在 PRD 中說明這與台灣醫療法規的關係**

### 9.3 修改 PRD 時

- 每次修改都需同步更新 frontmatter 的 `updated` 欄位與第 1 節 Change History
- 不得直接修改 status 為比當前更高的狀態（狀態升級需人工確認審核完成）
- 不得修改 `is_example: false` PRD 中的業務規則，除非有明確的變更說明

### 9.4 不應做的事

- 不得將 `is_example: true` 的範例 PRD 內容直接複製為正式需求
- 不得對涉及安全（如：電子簽章、存取控制、稽核日誌）的需求添加「可選」或「如有餘裕再做」的建議
- 不得假設某個外部系統（如 CDSS、健保署平台）的 API 行為，未知的介接規格應標示為待確認
- 不得省略非功能需求（效能、安全、可用性），即使 PM 沒有提到，也應提示補充

## 10. 常見跨系統業務規則

以下規則在多個系統的 PRD 中均適用，生成或修改 PRD 時若涉及相關情境，應確保與此處描述一致：

| 規則 ID | 規則描述 | 影響系統 |
|--------|---------|--------|
| X-BR-01 | 同一病患同一診別（門診/急診）同一日只能掛號一次，重複需人工二次確認 | Register |
| X-BR-02 | 診別判斷：過去 365 天內同科有就診紀錄者為複診，否則為初診 | Register, Billing |
| X-BR-03 | 重大傷病持有者（有效重大傷病卡）門診部分負擔歸零 | Billing |
| X-BR-04 | 醫令送出後不得刪除，更正需開立停用醫令或更正紀錄 | OPD, ER, IPD |
| X-BR-05 | 所有醫令需由開立醫師本人電子簽章，不接受共用帳號 | OPD, ER, IPD |
| X-BR-06 | 護理記錄送出後不得修改，更正需新建更正紀錄並說明原因 | ERNIS, IPDNIS |
| X-BR-07 | 病患個資操作（查詢、修改）均需寫入稽核日誌，包含操作者帳號與時間戳記 | 全系統 |
| X-BR-08 | CDSS 警示分三級：資訊（可略過）、警告（需確認）、危險（需填寫理由） | OPD, ER, IPD |
| X-BR-09 | 急診一、二級檢傷送出後，5 分鐘內未有醫師回應需觸發再次通知 | ERNIS |
| X-BR-10 | 健保申報同一月份不得重複送件，補申報需獨立流程並留存原送件紀錄 | Declare |
